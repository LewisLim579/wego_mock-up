# 주문자동화 비즈니스 프로세스 정의서 (v0.2)

> 본 문서는 주문자동화 비즈니스 프로세스를 작도(BPMN 등)할 수 있을 정도의 상세도로 정의한다.
> 각 단계는 액터(Actor) / 트리거(Trigger) / 입력(Input) / 활동(Activity) / 출력(Output) / 다음 단계(Next) / 예외(Exception) 구조로 기술한다.
> 첨부된 시스템 구성도(2026-04-28)를 기준으로 작성되었으며, 화면설계서(`주문자동화_화면설계.md`)와 함께 사용한다.

- 시스템 : `SKD AX Lab` (분석/추천) + `wego+` (운영) + `스마트미터` (현장 IoT)
- 작성일 : 2026-04-28

---

## 1. 프로세스 목적 & 범위

### 1-1. 목적
- 잔량 기반 AI 추천으로 적기 주문을 보장하고 긴급수송을 최소화한다.
- 주문 발의 → 주문 확정 → 배차 → 납품 → 정산 전 과정을 표준화한다.
- 사람이 손대는 영역을 “예외만”으로 축소한다.

### 1-2. 범위 (In-Scope)
- 추천 주문 생성 / 검토 / 확정
- 보류 수요처 관리 / 예외 처리
- 자차·책임수송 분기, 배차, 납품 등록, 마감 및 수송비 정산
- 스마트미터 이상 징후 모니터링 및 사용 상태 자동 판단

### 1-3. 범위 외 (Out-of-Scope)
- 채권/세금계산서 처리(정산 산출물만 연계)
- 차량 관제 시스템 내부(배차 결과만 수신)

## 5. End-to-End 프로세스 (메인 플로우)

> 이하 표기 규칙
> - **Actor / System** = 누가 수행하는가
> - **Trigger** = 단계의 시작 조건
> - **Input / Output** = 객체 또는 데이터
> - **Branch** = 분기 조건
> - **Next** = 다음 단계 ID
> - **Exception** = 발생 가능한 예외 → 분기 단계 ID

### 5-1. Process Map (단계 ID 일람)

```
P0  데이터 수집
P1  데이터 송신 (wego+ → AX Lab)
P2  분석 / 모델 실행
P3  추천/이상징후 회신 (AX Lab → wego+)
P4  추천 주문 List 표시
P5  추천 검토 (담당자)
P6  주문 확정 / 보류 / 수기 / 고정 분기
P7  주문신청 저장
P8  분기 결정 (자차 vs 책임수송)
P9a 자차 배차 → 자차기사 운행
P9b 책임 배차(CHL/2차) → 책임기사 운행
P10 납품 등록
P11 마감 및 수송비 정산
P12 결과 환원 (주문 생성 이벤트 → 추천 학습)
```

---

### P0. 데이터 수집

| 항목 | 내용 |
|---|---|
| Actor/System | S3 스마트미터 → S2 wego+ |
| Trigger | 샘플링 주기 도달(15/30/60min) 또는 이벤트 |
| Input | 잔량(%/kg), 단말 상태(배터리·통신) |
| Activity | 단말 송신 → wego+ 수신 → DB 적재, 이상치 1차 검증 |
| Output | 잔량 시계열, 단말 이벤트 |
| Next | P1 |
| Exception | 통신 끊김(n시간) → S08 모니터링에 “수신 끊김”으로 등록 |

### P1. 데이터 송신 (wego+ → AX Lab)

| 항목 | 내용 |
|---|---|
| Actor/System | S2 wego+ |
| Trigger | 일배치(예: 매일 02:00) + 이벤트(이상징후 발생 즉시) |
| Input | 수요처/탱크 정보, 납품 실적, 잔량 데이터 |
| Activity | wego+ → AX Lab API 송신 |
| Output | AX Lab 기준 데이터 갱신 |
| Next | P2 |
| Exception | API 실패 → 재시도 3회, 실패 시 운영관리자(A3) 알림 |

### P2. 분석 / 모델 실행

| 항목 | 내용 |
|---|---|
| Actor/System | S1 SKD AX Lab |
| Trigger | P1 완료 / 수동 재실행(A3) |
| Input | 기준 데이터 + 외생변수(기상·캘린더, S4) |
| Activity | (a) 데이터 정제·파생변수 생성 (b) 군집/예측 모델 실행 (c) 안전재고 임계 비교 (d) 이상 패턴 분류 |
| Output | 주문 추천 결과(수요처별), 스마트미터 이상 징후 이벤트 |
| Next | P3 |
| Exception | 데이터 부족(SM 미설치/관측<30일) → 군집 평균 보조 예측, 신뢰도 하향 |

### P3. 추천/이상징후 회신

| 항목 | 내용 |
|---|---|
| Actor/System | S1 → S2 (API) |
| Trigger | P2 완료 |
| Input | 주문 추천, 이상 징후 |
| Activity | wego+ API 수신 → “주문 자동화 List” 갱신, “스마트미터 이상 징후” 큐 적재 |
| Output | 사용자 화면용 데이터 |
| Next | P4 |
| Exception | API 수신 실패 → 직전 결과 유지, 재시도 |

### P4. 추천 주문 List 표시

| 항목 | 내용 |
|---|---|
| Actor/System | S2 wego+ |
| Trigger | 데이터 갱신 / 화면 진입 |
| Input | 추천 결과 + 보류 풀 |
| Activity | 우선순위(P0/P1/P2) 정렬, 보류 수요처 분리 표시, 거래처 발의 B/L 신청과 통합 노출 |
| Output | 사용자 화면(S01/S02), 모바일 푸시(긴급) |
| Next | P5 |
| Exception | 보류 중 만료 도래 → 일반 풀로 자동 복귀 |

### P5. 추천 검토 (담당자)

| 항목 | 내용 |
|---|---|
| Actor | A2 충전소 담당자 |
| Trigger | 출근 시 알림 / 시간대 확인 |
| Input | 추천 List, 결정 근거(예측·임계·신뢰도) |
| Activity | 우선순위 순으로 검토, 권장량/일시 확인, 차량 가용성 확인 |
| Output | 확정 / 보류 / 수기수정 의사결정 |
| Next | P6 |
| Exception | 모델 신뢰도 < 70% → 결정 근거 추가 확인, 필요시 A3 운영관리자에 보고 |

### P6. 주문 분기 결정

P5 결과에 따라 4개 트랙으로 분기한다.

#### P6-A. 추천 확정
| 항목 | 내용 |
|---|---|
| Trigger | 담당자 [확정] |
| Activity | 주문량/납품일/차량 확인 후 확정 |
| Output | 주문 객체(생성유형=추천) |
| Next | P7 |

#### P6-B. 보류 처리
| 항목 | 내용 |
|---|---|
| Trigger | 담당자 [보류] + 사유 선택 (다시알림/거래중단/비활성/누락/거래종료) |
| Activity | 보류 수요처 풀로 이동, 사유·기간·메모 기록 |
| Output | 보류 객체 (만료일·재추천일 보유) |
| Next | (만료 시) P4 재진입, (영구 종료) 종료 |

#### P6-C. 수기 주문
| 항목 | 내용 |
|---|---|
| Trigger | 외부 채널(전화/팩스) 수신 후 [수기 접수] |
| Activity | 수요처/주문량/납품일 직접 입력 |
| Output | 주문 객체(생성유형=수기) |
| Next | P7 |
| Note | 모델 학습 시 해당 주문은 “수기” 플래그로 별도 처리 가능 |

#### P6-D. 고정 주문
| 항목 | 내용 |
|---|---|
| Trigger | 사전 정기 스케줄 도달 |
| Activity | 시스템이 고정 주문 자동 생성 |
| Output | 주문 객체(생성유형=고정) |
| Next | P7 |

> P6에서는 동일 수요처에 대해 추천+B/L신청이 동시 존재할 수 있다. 이 경우 동일 일자 합치(merge) 룰을 적용 → 자세한 룰은 §7-3.

### P7. 주문신청 저장

| 항목 | 내용 |
|---|---|
| Actor/System | A2 / S2 wego+ |
| Trigger | P6-A/C/D 완료 |
| Input | 주문 객체(생성유형 포함) |
| Activity | 정합성 검증(임계·차량·시간) → 저장 → 상태 `saved` |
| Output | 저장된 주문 |
| Next | P8 |
| Exception | 차량/시간 충돌 → P5로 복귀, 사용자 재선택 |

### P8. 분기 결정 (자차 vs 책임수송)

| 항목 | 내용 |
|---|---|
| Actor/System | S2 wego+ (룰 자동) + A2 확인 |
| Trigger | P7 완료 |
| Input | 수요처 운송정책, 차량 가용성 |
| Activity | 분기 룰 적용 (§7-1) |
| Output | 주문에 분기 코드(자차/책임) 부여 |
| Next | 자차 → P9a, 책임 → P9b |
| Exception | 분기 룰 미정의 → A3 운영관리자에게 결정 요청 |

### P9a. 자차 배차 → 운행

| 항목 | 내용 |
|---|---|
| Actor | A1 거래처 → A4 자차기사 |
| Trigger | 분기=자차 |
| Activity | 거래처가 자차 차량 배차 → 자차기사 운행 |
| Output | 배차 객체, 상태 `dispatched` |
| Next | P10 |
| Exception | 자차 가용 차량 없음 → 책임수송으로 변경 요청 → P9b 분기 변경 |

### P9b. 책임 배차(CHL/2차) → 운행

| 항목 | 내용 |
|---|---|
| Actor | S2 wego+ (배차 모듈) → A5 책임기사 (CHL 또는 2차 수송사) |
| Trigger | 분기=책임 |
| Activity | CHL/2차 수송사 배차 → 책임기사 운행 |
| Output | 배차 객체, 상태 `dispatched` |
| Next | P10 |
| Exception | CAPA 초과 → 시점 후행 조정(D+1) 또는 분할 |

### P10. 납품 등록

| 항목 | 내용 |
|---|---|
| Actor | A4 자차기사 / A5 책임기사 |
| Trigger | 현장 납품 완료 |
| Input | 실납품량, 납품 시각, 사진/메모 |
| Activity | 모바일에서 납품 결과 등록 |
| Output | 납품 객체, 상태 `delivered` |
| Next | (자차) P12, (책임) P11 |
| Exception | 부분 납품 / 반품 → 납품 보정 입력 + A2 승인 |

### P11. 마감 및 수송비 정산

| 항목 | 내용 |
|---|---|
| Actor/System | S2 wego+ + S4 외부 회계 연계 |
| Trigger | 책임수송 분기 + 마감 주기(일/주/월) 도래 |
| Input | 책임수송 납품 실적, 수송비 단가 정책 |
| Activity | 수송비 산정 → 정산 데이터 생성 → 외부 회계 연계 |
| Output | 정산 객체, 상태 `settled` |
| Next | P12 |
| Exception | 단가 미정의 / 분쟁 → A3 보류 처리 |

### P12. 결과 환원 (학습 루프)

| 항목 | 내용 |
|---|---|
| Actor/System | S2 wego+ → S1 AX Lab |
| Trigger | 납품/정산 완료 시 “주문 생성” 이벤트 |
| Activity | 실납품량·적중여부·사유 등을 학습 데이터로 환원, 추천 풀 갱신 |
| Output | 다음 P2 회차의 입력 |
| Next | (다음 회차) P1 |
| Exception | 데이터 정합성 오류 → 학습 제외 플래그 |

## 6. 서브 프로세스

### 6-1. 보류 수요처 관리 서브 프로세스

```
[P6-B 보류 발생] ──► (보류 풀 등록) ──► [만료 도달?]
                                          ├── 예 ──► (자동 복귀) ──► P4
                                          └── 아니오 ──► (모니터링 유지)
                                                         │
                                                         ├── 사유 변경 / 만료일 연장 (A3)
```

| 수기 접수 | 즉시(주문 생성과 동시) | 수기 플래그 |
| 거래 종료 | 무기한 | 영구 제외 |

## 7. 비즈니스 룰 (Decision / Business Rules)

### 7-1. 자차 vs 책임수송 분기 룰 (P8)

| 우선순위 | 조건 | 결과 |
|---|---|---|
| 1 | 수요처 마스터에 “자차 전용” 플래그 | 자차 |
| 2 | 거래처 마스터에 “자차 우선” + 자차 가용 차량 존재 | 자차 |
| 3 | 자차 가용 차량 없음 또는 시간대 충돌 | 책임수송 |
| 4 | 그 외 | 책임수송 |

### 7-2. 우선순위 산정 룰 (P4)

| 우선순위 | 조건 |
|---|---|
| P0 (긴급) | 잔량 < 20% 또는 D+0 소진 예상 |
| P1 (주의) | 잔량 20~30% 또는 D+1 소진 예상 |
| P2 (정상) | 잔량 > 30% 또는 D+2~3 소진 예상 |

### 7-3. 동일 수요처 중복 주문 합치 룰 (P6/P7)

| 조건 | 처리 |
|---|---|
| 추천 + B/L 신청 동일 일자 | 권장량 vs 신청량 비교 → 더 큰 값으로 1건 통합 |
| 추천 + 고정 동일 일자 | 고정 우선, 추천은 자동 보류 |
| 추천 + 수기 동일 일자 | 수기 우선, 추천은 자동 보류 |
| 동일 수요처 분할 권고 | 1차/2차 별도 주문 객체로 분리 |

### 7-4. 이상 징후 처리 룰 (S08 / 6-2)

| 이벤트 | 심각도 | 자동 조치 | 사람 조치 |
|---|---|---|---|
| 납품 중 잔량 무변동 | High | 샘플링↑, 알림 | 현장 점검 |
| 납품 없는데 잔량 급증 | High | 검증 플래그 | 데이터 보정 |
| 수신 끊김(n시간) | Mid | 통신 이상 표시 | 통신사 확인 |
| 3일 연속 사용 0kg | Mid | 휴업 추정 → 보류 후보 | 보류 처리 |
| 이상 과다/저하 소비 | Mid | 패턴 분류 + 알림 | 사유 확인 |
| 데이터 정합성 오류 | Low | 학습 제외 | 누락 처리 |

### 7-5. 안전재고 임계 룰

| 항목 | 룰 |
|---|---|
| Hard Fail | 잔량 ≤ 15% (절대 도달 금지) |
| 동적 임계(Dynamic SS) | `Hard Fail + (Daily Max × 리드타임 + σ버퍼)` 로 산출 |
| 임계 도달 | 시스템이 P0 우선순위로 추천 강제 생성 |

### 7-6. 모델 신뢰도 라벨

| 신뢰도 | 라벨 | UI 처리 |
|---|---|---|
| ≥85% | 높음(Green) | 일반 표시 |
| 70~84% | 보통(Yellow) | 결정 근거 강조 |
| <70% | 낮음(Red) | 사용자에게 “수기 검증 권고” |

## 8. 상태 머신 (Order State Machine)

```
generated ──► reviewed ──► saved ──► dispatched ──► delivered ──► settled
   │             │           │
   │             │           ├──► canceled
   │             │           └──► manually_modified
   │             │
   ├──► held ──► (만료) ──► generated 재진입
   │
   └──► excluded
```

| 상태 | 진입 조건 | 다음 가능 상태 |
|---|---|---|
| generated | AX Lab 추천 생성 | reviewed / held / excluded |
| reviewed | 담당자 검토 시작 | saved / held |
| saved | 주문신청 저장 완료 | dispatched / canceled / manually_modified |
| held | 보류 처리 | generated(재추천) / excluded |
| excluded | 영구 제외 | (종착) |
| dispatched | 배차 완료 | delivered / canceled |
| delivered | 납품 등록 완료 | settled (책임수송) / 종착 (자차) |
| settled | 정산 완료 | (종착) |
| canceled | 취소 처리 | (종착) |

## 14. 가정 / 제약

- 모든 상태 변경은 감사 로그 보존(누가/언제/무엇을/사유)
- AX Lab은 결정권을 갖지 않으며, 최종 확정은 항상 A2(사람)
- 보류는 추천 풀에서 분리되어 모델 입력에서도 별도 취급
- 수기 주문은 신뢰도 평가 대상에서 제외하지만 정산에는 포함
- 외생변수 결측 시 모델은 군집 평균으로 대체(보조 예측)
